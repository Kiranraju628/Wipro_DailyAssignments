Module 1: Understanding Circuit Breaker Concept
===============================================

1. What is Cascade Failure?

A cascade failure occurs when one microservice fails and that failure spreads to other dependent services, causing the entire system to become unstable.

Example
-------

Service A → Service B → Service C
                      ↓
                   Failure
                      ↓
Service B waits indefinitely
                      ↓
Service A also fails


How Circuit Breaker Prevents System Collapse?

A Circuit Breaker detects repeated failures and stops sending requests to the failed service temporarily.

Instead of waiting for a timeout:
---------------------------------

Request
   ↓
Circuit Breaker
   ↓
Fallback Response


This prevents:

Resource exhaustion
Long response times
Cascade failures


Circuit Breaker Diagram
-----------------------

Without Circuit Breaker
-----------------------

Client
  |
Service A
  |
Service B
  |
Service C (DOWN)
  |
Failure propagates


With Circuit Breaker
--------------------

Client
  |
Service A
  |
Circuit Breaker
  |
Service B
  |
Service C (DOWN)

Circuit Opens
     ↓
Fallback Response



Assignment Questions
--------------------

What are the 3 States of a Circuit Breaker?
1. CLOSED State
Normal State
Requests pass normally.
Failures are monitored.

2. OPEN State
Too many failures detected
Requests are blocked.
Fallback method executes.

3. HALF-OPEN State
Testing State
Allows a few requests.
If successful → CLOSED.
If failed → OPEN.


Retry Pattern vs Circuit Breaker Pattern
----------------------------------------

| Feature          | Retry Pattern        | Circuit Breaker Pattern |
| ---------------- | -------------------- | ----------------------- |
| Purpose          | Retry failed request | Stop failed requests    |
| Action           | Attempts again       | Blocks calls            |
| Network Traffic  | Increased            | Reduced                 |
| Failure Handling | Temporary issues     | Continuous failures     |
| Example          | Retry 3 times        | Open circuit            |

Task 3: Enable Circuit Breaker
------------------------------

@Service
public class OrderService {

    @Autowired
    private RestTemplate restTemplate;

    @CircuitBreaker(
            name = "paymentService",
            fallbackMethod = "paymentFallback")
    public String getPayment(String orderId) {

        return restTemplate.getForObject(
                "http://localhost:8081/payment/"
                        + orderId,
                String.class);
    }

    public String paymentFallback(
            String orderId,
            Throwable t) {

        return "Payment service is currently unavailable. Please try later.";
    }
}

Module 4: Circuit Breaker Configuration
=======================================

application.yml
---------------

resilience4j:
  circuitbreaker:
    instances:
      paymentService:
        registerHealthIndicator: true
        slidingWindowSize: 5
        failureRateThreshold: 50
        waitDurationInOpenState: 10s
        permittedNumberOfCallsInHalfOpenState: 3


What is failureRateThreshold?
failureRateThreshold = 50

Meaning:

If 50% or more requests fail in the configured window, the circuit opens.

Example
-------

Sliding Window = 5 requests

Success = 2
Failure = 3

Failure Rate = 60%

Circuit → OPEN


What Happens in HALF-OPEN State?

After the OPEN timeout expires:

OPEN
  ↓
HALF-OPEN

Allows a limited number of requests.

Example:

permittedNumberOfCallsInHalfOpenState = 3

If all succeed:

HALF-OPEN → CLOSED

If any fail:

HALF-OPEN → OPEN

Questions
---------

Why Use Config Server?

Benefits:

Centralized configuration management
Avoid duplicate configurations
Dynamic configuration updates
Environment-specific configurations
Better security and maintainability
Example

Without Config Server:

Order Service -> application.yml
Payment Service -> application.yml
User Service -> application.yml

With Config Server:

Config Server
      |
--------------------------------
| Order | Payment | User |
--------------------------------


Difference Between application.yml and bootstrap.yml
----------------------------------------------------

| Feature              | application.yml      | bootstrap.yml                  |
| -------------------- | -------------------- | ------------------------------ |
| Purpose              | Application settings | External configuration loading |
| Load Time            | After bootstrap      | Before application startup     |
| Config Server Access | No                   | Yes                            |
| Priority             | Lower                | Higher                         |
| Usage                | Business properties  | Config Server connection       |
